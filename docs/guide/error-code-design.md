# 错误码方案设计

## 1. 背景

当前系统已经具备 `code`、`reason`、`message` 等错误语义基础，但在演进过程中曾出现以下问题：

- 业务错误与系统错误边界不清，部分本应提示给前端的错误被当作 `500` 返回；
- 错误定义方式并存，历史上同时存在主错误模型与兼容错误模型；
- HTTP、gRPC、日志与观测未完全共享同一套稳定语义；
- `Reason` 命名与服务级错误文案风格存在漂移；
- 一些过度设计的抽象增加了理解与维护成本。

本方案的目标是在**保持现有前端返回结构兼容**的前提下，建立一套统一、稳定、可维护的错误码体系，并将当前已经确认的收缩与清理结论固化为唯一权威文档。

---

## 2. 目标与范围

### 2.1 目标

本方案遵循以下目标：

- **前端友好**：前端能够稳定依据 `code` 与 `reason` 做提示、分支判断和埋点统计；
- **协议清晰**：HTTP 状态码只承担协议层和系统层语义，不再与业务语义混用；
- **统一语义**：HTTP、gRPC、日志与监控共享同一套错误定义；
- **兼容优先**：不破坏现有响应体主结构，优先降低改造风险；
- **简单可维护**：删除无消费价值的抽象，收敛为最小可用核心模型。

### 2.2 适用范围

本方案适用于以下范围：

- 对外 HTTP API；
- 内部或对外 gRPC 接口；
- 服务日志与错误排障；
- 指标、告警与链路观测；
- 前后端错误处理约定。

### 2.3 非目标

本方案不包含以下内容：

- 多语言文案中心建设；
- 全量历史错误的一次性重命名；
- 与当前版本无关的错误治理平台建设；
- 重做整套数字错误码编码规则；
- 与当前错误码主题无关的全局架构重构。

---

## 3. 统一错误模型

### 3.1 唯一主模型

当前系统以 `BizError` 作为**唯一主错误模型**。

`BizError` 的核心字段包括：

- `Code`：稳定的数字业务码，用于程序分支判断、埋点统计与报表聚合；
- `Reason`：稳定的字符串错误标识，用于程序分支、日志检索与告警聚类；
- `Message`：默认用户提示文案；
- `HTTPStatus`：协议层返回语义；
- `Details`：补充错误详情；
- `Metadata`：补充上下文元数据；
- `Level`：内部派生的错误级别字段，用于实现辅助，不作为对外主要契约。

其中：

- `Code` 负责程序分支判断；
- `Reason` 负责稳定标识与检索；
- `Message` 负责用户可读表达；
- `HTTPStatus` 负责协议层返回语义；
- `Details` 与 `Metadata` 继续承担补充上下文的职责。

### 3.2 已删除的旧模型与抽象

以下设计已经移除，不再属于当前错误码体系：

- `catalog`；
- `LookupSpec`；
- `LookupSpecByReason`；
- `Kind`；
- `Module`；
- `Retryable`；
- `ErrorXCompat`；
- `NewCompat`；
- `GetModule()`。

当前系统不再保留第二套错误承载模型，也不再维护注册表式错误查找抽象。

### 3.3 响应体兼容策略

对外继续兼容现有响应主结构：

- 成功：`code = 0`；
- 失败：保留 `code`、`reason`、`message`，失败时 `data` 为空或省略。

建议对外语义如下：

- `code`：数字业务码；
- `reason`：稳定字符串标识；
- `message`：默认展示文案；
- `data`：成功数据载荷，失败时不承载错误详情。

### 3.4 文案策略

后端返回默认 `message`，前端可以：

- 直接展示后端文案；
- 按 `code` 或 `reason` 覆盖成本地化文案；
- 在不同端上做差异化展示。

推荐约定：

- 前端稳定依赖 `code` 与 `reason`；
- `message` 用于默认展示和兜底文案；
- 不依赖 HTTP `200` 与否去判断具体业务失败原因。

### 3.5 敏感信息控制

面向前端的 `message` 不应直接暴露以下内容：

- SQL 语句；
- 数据库原始报错；
- 内部堆栈；
- 服务内部地址；
- 不适合终端用户看到的基础设施细节。

内部排障信息应进入日志、`details` 或 `metadata`，并根据环境进行控制。

## 4. HTTP 语义规则

### 4.1 当前原则

系统采用四层错误语义：

1. **HTTP Status**：表达协议层、入口层与系统层语义；
2. **Code**：稳定数字业务码；
3. **Reason**：稳定字符串错误标识；
4. **Message**：默认用户提示文案。

设计原则如下：

- `HTTP Status` 不再承担完整业务语义；
- `Code` 与 `Reason` 作为调用方稳定依赖的双主键；
- `Message` 由后端提供默认值，但不强制前端原样展示；
- 错误处理统一以当前错误模型为准，不依赖历史兼容层或注册表推导。

### 4.2 协议层错误

协议层错误直接返回 `4xx`，典型场景包括：

- 路由不存在；
- API 版本不存在或不受支持；
- HTTP 方法不允许；
- 请求头缺失；
- 内容类型错误；
- 请求体格式非法；
- 参数无法解析；
- 基础协议校验失败。

该类错误的核心特征是：**请求未能以正确协议语义进入业务处理阶段**。

### 4.3 业务层错误

业务层错误统一返回 `200`，通过响应体中的 `code`、`reason`、`message` 表达失败，典型场景包括：

- 用户已存在；
- 用户不存在；
- 密码错误；
- 余额不足；
- 资源状态不允许；
- 业务规则校验失败；
- 正常业务流程中的拒绝性结果。

该类错误的核心特征是：**请求本身合法，系统也正常工作，但业务上不满足成功条件**。

### 4.4 系统层错误

系统层错误返回 `5xx`，典型场景包括：

- 数据库连接失败、读写失败；
- 缓存不可用；
- 下游服务超时、不可用或返回不可恢复异常；
- 代码 panic、未知异常；
- 配置缺失或内部依赖初始化失败。

该类错误的核心特征是：**调用方通常无法自行修复，需要开发、运维或基础设施介入**。

### 4.5 认证授权特殊规则

认证与授权采用单独边界：

- `token` 缺失、非法、过期、签名错误：返回 `401`；
- 已认证但无权限访问资源或执行操作：返回 `403`；
- 普通业务失败仍返回 `200`。

### 4.6 HTTP 状态码来源

当前 `BizError.HTTPStatus()` 的语义统一**基于 `Code` 推导**。

这样做的目的：

- 收口 HTTP 语义来源；
- 避免同一错误语义在多个位置重复定义；
- 降低错误定义与 HTTP 行为漂移的概率。

### 4.7 局部显式例外

极少数不适合通过统一错误码映射表达的协议层语义，可以在局部显式表达。

例如：

- `NoRoute` 的 `404` 由路由层显式返回；
- 不再通过全局注册表维持此类特例。

原则是：

- 例外必须局部、显式、低抽象；
- 不能为少量特例恢复整套全局注册表设计。

## 5. Reason 命名规则

### 5.1 统一规则

`Reason` 统一采用以下形式：

`<Domain>.<Action>`

要求：

- 每一段均使用 PascalCase；
- 命名应表达稳定语义，而不是偶然实现细节；
- 不再新增单段旧风格 `Reason`。

### 5.2 合规示例

- `Common.InternalError`；
- `Common.NotFound`；
- `Common.InvalidArgument`；
- `Auth.TokenInvalid`；
- `Auth.TokenExpired`；
- `User.AlreadyExists`；
- `User.NotFound`；
- `Infra.DatabaseReadFailed`；
- `Service.Unavailable`。

### 5.3 收敛原则

- 保留当前已合规的 `Auth.*`、`Infra.*`、`User.*`、`Service.*` 风格；
- 将历史单段 `Reason` 收敛为命名空间风格；
- 避免把技术实现细节直接暴露为协议层 `Reason`；
- 缩写或技术性前缀不再单独保留为特殊命名风格。

### 5.4 不再允许的旧风格

以下风格不再允许新增：

- 单段 `Reason`，如 `InternalError`、`NotFound`、`BindError`；
- 只为了兼容历史测试而保留的技术型命名；
- 没有领域前缀的兜底字符串。

## 6. 服务级错误目录约定

### 6.1 `internal/pkg/errno` 的定位

`internal/pkg/errno` 作为本服务的错误字典层，负责：

- 按领域维护预定义错误；
- 暴露稳定的错误变量或构造入口；
- 作为前端、业务层与中间件共享的服务级错误目录。

该层不再承担历史兼容模型的过渡职责。

### 6.2 当前约定

`internal/pkg/errno` 中的错误定义应遵循以下规则：

- 通用错误使用 `Common.*` 风格 `Reason`；
- 认证错误使用 `Auth.*`；
- 基础设施错误使用 `Infra.*`；
- 服务状态错误使用 `Service.*`；
- 用户域错误使用 `User.*`；
- `Message` 默认使用简体中文。

### 6.3 `code.go` 约定

`internal/pkg/errno/code.go` 中的通用错误应统一为：

- `Common.InternalError`；
- `Common.NotFound`；
- `Common.BindError`；
- `Common.InvalidArgument`；
- `Common.OperationFailed`；
- `Common.PageNotFound`。

### 6.4 `user.go` 约定

`internal/pkg/errno/user.go` 中的用户域错误保持 `User.*` 风格 `Reason`，并统一使用中文 `Message`。

### 6.5 文案规范

`Message` 统一遵循以下规则：

- 使用简体中文；
- 保持用户可读、简洁直接；
- 中文句末建议统一使用 `。`；
- 不暴露内部实现细节。

## 7. 错误码治理规则

### 7.1 分段管理

错误码采用按模块分段治理，保留五位编号风格，示例分段如下：

- `10xxx`：通用与入口层；
- `20xxx`：认证授权；
- `21xxx`：用户域；
- `90xxx`：数据库、缓存、外部依赖；
- `99xxx`：未知内部错误。

说明：

- 分段用于治理和协作，不作为运行时自动推导全部 HTTP 语义的唯一依据；
- 已经对外暴露且调用方可能依赖的错误码，在当前版本范围内应尽量保持稳定；
- 新增错误必须在所属模块码段内分配。

### 7.2 唯一性规则

必须满足以下治理约束：

- 一个 `Code` 只能对应一个 `Reason`；
- 一个 `Reason` 只能对应一个 `Code`；
- 一个错误语义只能有一个标准定义；
- 不允许直接使用 HTTP 状态码充当业务错误码；
- 不允许多个业务语义复用同一个错误码。

---

## 8. gRPC、日志与观测约定

### 8.1 gRPC 对齐

gRPC 继续使用标准 `status code` 表达协议语义，同时在错误详情中补充：

- `reason`；
- `code`；
- 必要元数据。

HTTP 与 gRPC 共享同一套错误定义，仅出口协议适配不同。

### 8.2 日志字段

发生错误时，日志建议至少记录：

- `traceID`；
- `requestID`；
- `userID`（如有）；
- `httpStatus`；
- `bizCode`；
- `reason`；
- `path`；
- `method`。

当前日志与观测不再依赖 `module`、`kind`、`retryable` 这类已被删除的历史字段。

### 8.3 指标与告警

指标与告警建议优先以以下维度聚合：

- `http_status`；
- `reason`；
- `biz_code`。

---

## 9. 已删除与禁用的旧设计

以下设计已被明确移除或禁用，不再允许继续使用：

- `catalog`；
- `LookupSpec`；
- `LookupSpecByReason`；
- `Kind`；
- `Module`；
- `Retryable`；
- `ErrorXCompat`；
- `NewCompat`；
- `GetModule()`；
- 旧单段 `Reason` 风格；
- 在 `errno` 层继续使用英文默认 `Message`。

新代码与新文档都应以当前收缩后的规则为准，不再继续扩散这些历史设计。

---

## 10. 测试、迁移与验收原则

### 10.1 测试原则

测试应从“验证抽象存在”转为“验证行为正确”。

重点验证：

1. `BizError` 创建结果是否稳定；
2. `HTTPStatus()` 是否按 `Code` 返回正确语义；
3. `Code()` 与 `Reason()` 提取行为是否稳定；
4. `FromError()` 是否保留核心错误信息；
5. `errno` 预定义错误是否仍保持对外稳定；
6. 服务级错误定义的 `Reason` 与 `Message` 是否符合当前规范。

### 10.2 迁移原则

后续错误码演进遵循以下原则：

- 先收口新增，再治理历史；
- 优先删除无实际消费价值的抽象；
- 优先保持 `Code` 与 `HTTPStatus` 语义稳定；
- 对个别例外语义，优先选择局部、显式、低抽象表达；
- 不顺手扩大到与当前主题无关的全局重构。

### 10.3 验收标准

当前错误码体系的验收标准如下：

- `docs/guide/error-code-design.md` 是唯一权威设计文档；
- 系统统一以 `BizError` 作为唯一主错误模型；
- `HTTPStatus`、`Code`、`Reason`、`Message` 的职责边界清晰一致；
- `Reason` 命名规则统一，不再混用旧单段风格；
- `internal/pkg/errno` 的服务级错误目录符合当前风格约束；
- 已删除与禁用的旧设计不再在代码或文档中作为当前方案出现；
- `go test ./...` 通过。

---

## 11. 最终结论

当前错误码体系采用**单一主模型 + 明确协议边界 + 统一命名规则 + 服务级目录约定**的设计。

简而言之：

- 对外保留兼容响应结构；
- 对内收敛为 `BizError` 主路径；
- 以 `HTTPStatus + Code + Reason + Message` 形成稳定错误语义；
- 以 `4xx / 200 / 5xx` 明确协议错误、业务错误、系统错误边界；
- 以 `Reason` 统一命名规则和 `errno` 层中文文案保障长期可维护性。
